# Execution, subagents, and isolation

Cross-references: [04 Verification Gates](./04-verification-gates.md) consumes subagent completions and evidence; [05 Memory, Learning, Continuation](./05-memory-learning-continuation.md) promotes verified execution outcomes into reusable memory.

## Spec

Mission Control execution should be a clean chain:

1. `MissionTaskDispatcher` selects mission tasks and invokes a runner per task.
2. `MissionTaskRunner` binds the subagent contract to `{ missionId, taskId }`, delegates exactly one executor run, and derives durable evidence refs from the executor result.
3. `SubagentContract` is the boundary object for role, file scope, success criteria, escalation policy, required outputs, mission binding, and parent revision freshness.
4. Mutation tools enforce the active contract or mission scope before writes. The prompt is advisory; tool-level scope enforcement is authoritative.
5. Worktree isolation materializes a per-task merged directory, captures task deltas, and can commit task-only changes to `amaze/task/<taskId>` branches for later replay.
6. The dispatcher records success/failure attempts and appends runner evidence refs to the mission evidence list so verification and learning can cite concrete artifacts.

The planner should treat execution tasks as narrow contracts, not free-form prompts. A generated Mission Control plan should include: task id, target files, success criteria, scope include/exclude globs, escalation policy, isolation mode, and expected evidence refs.

## Source Evidence

Its default runner intentionally throws, so production wiring must inject a real runner factory.
- `src/task/mission-task-runner.ts`: `MissionTaskRunner` stores `{ missionId, taskId }`, binds `options.contract` through `bindContractToMission` and `enforceMissionBinding`, delegates to `runSubprocess`, derives `task-output://`, `task-patch://`, `task-branch://`, and `task-run://` evidence refs, and returns a new `MissionTask` snapshot rather than persisting.
- `src/task/worktree.ts`: isolation resolves PAL backends from task isolation modes, creates `mergedDir` under the encoded project worktree directory, cleans isolation in `cleanupIsolation`, captures baseline/delta patches including nested repos, commits root changes to `amaze/task/<taskId>`, and merges branches sequentially with conflict reporting.
- `src/subagent/contract.ts`: `SubagentContract` defines mission binding, parent revision, scope, success criteria, escalation, optional input artifact, and must-produce outputs. It renders deterministically to XML, detects stale contracts, verifies subagent completions, derives child scope from parent mission scope, and enforces contract, goal, and mission scopes.
- `src/subagent/mutation-scope.ts`: `enforceMutationScope` resolves real mutation targets under session cwd, rejects paths outside cwd, then applies single precedence: active `SubagentContract` first; otherwise active Mission scope.
- `src/tools/write.ts`: `WriteTool` calls `enforceMutationScope` before archive writes, SQLite row writes, and normal file writes, then separately applies plan-mode and autogenerated-file guards.

## Execution Model

### Dispatcher responsibilities

The dispatcher is orchestration glue only. It should not decide file scope, isolation backend, verifier policy, or learning policy. It should:

- run the mission's selected tasks in declared order unless a higher-level scheduler partitions them;
- pass the mission task to the runner factory so the factory can bind mission/task ids;
- collect evidence refs into `mission.evidenceRefs`;
- record task attempt events with success/failure notes;
- distinguish blocked tasks from failed tasks.

### Runner responsibilities

The runner is a single-run adapter. It should:

- bind `SubagentContract.missionId` and `SubagentContract.taskId` when a mission context exists;
- fail closed if the contract contradicts the expected mission/task binding;
- preserve executor behavior rather than reimplementing subprocess, fan-out, or worktree semantics;
- return evidence refs and a task snapshot; persistence remains the caller's responsibility.

### Contract and scope responsibilities

A Mission Control planner should mint one subagent contract per bounded deliverable. The contract should include the smallest include scope that can satisfy the task and all parent-denied paths in exclude scope. A delegated child may narrow the parent scope but must never exceed it. Mutation tools must consult structural guards before every create/update/delete path.

### Isolation responsibilities

Isolation is execution infrastructure, not a permission system. It prevents accidental cross-task working-tree coupling and provides clean patch/branch evidence, while contract and mission scope still decide whether a mutation is allowed.

Use isolation evidence as follows:

- `task-output://...` for the subagent transcript/output artifact;
- `task-patch://...` for captured patch evidence;
- `task-branch://amaze/task/<taskId>` when changes were committed to a task branch;
- `task-run://<id>` as the stable run handle even when no file artifact exists.

## Target TypeScript Sample: default runner wiring

```ts
// Target TypeScript sample; not current implementation.
import { MissionTaskDispatcher } from "../mission/core/mission-task-dispatcher";
import type { MissionTask } from "../mission/core/mission-task";
import { MissionTaskRunner } from "../task/mission-task-runner";
import type { ExecutorOptions } from "../task/executor";

export function createMissionDispatcher(missionId: string): MissionTaskDispatcher {
	return new MissionTaskDispatcher((task: MissionTask) => {
		const runner = new MissionTaskRunner({ missionId, taskId: task.id });
		return {
			async run(options: ExecutorOptions, taskSeed: MissionTask) {
				return runner.run(
					{
						...options,
						contract: options.contract
							? {
									...options.contract,
									scope: taskSeed.scope ?? options.contract.scope,
									successCriteria: taskSeed.successCriteria ?? options.contract.successCriteria,
								}
							: undefined,
					},
					taskSeed,
				);
			},
		};
	});
}
```

## Target TypeScript Sample: scoped task execution

```ts
// Target TypeScript sample; not current implementation.
import type { MissionTask } from "../mission/core/mission-task";
import type { SubagentContract } from "../subagent/contract";

export function contractForTask(missionId: string, task: MissionTask, parentRev: number): SubagentContract {
	return {
		missionId,
		taskId: task.id,
		role: task.assignedAgent ?? "builder",
		parentMissionRev: parentRev,
		scope: task.scope ?? { include: [], exclude: [] },
		successCriteria: task.successCriteria ?? [],
		escalation: { onUncertainty: "ask-parent", budgetCap: 80_000 },
		outputContract: {
			mustProduce: [`task-run://${task.id}`],
		},
	};
}

export async function runScopedMissionTask(args: {
	dispatcher: MissionTaskDispatcher;
	mission: Mission;
	task: MissionTask;
}) {
	return args.dispatcher.run([args.task], {
		scopeGuard: args.mission.scopeGuard,
		evidenceRefs: args.mission.evidenceRefs,
		recordAttempt(taskId, attempt, note) {
			// Persist via MissionStore.recordTaskAttemptCheckpoint in production wiring.
			console.debug({ taskId, attempt, note });
		},
	});
}
```

## Mission Control Acceptance Criteria for Builders

- A mission execution plan emits one bounded `SubagentContract` per task with mission/task binding, scope, success criteria, escalation, and must-produce evidence.
- A default Mission Control runtime path injects a real `MissionTaskRunner` factory; the dispatcher default throwing runner is not reachable in production execution.
- Mutating tools reject writes outside the active subagent contract, parent mission scope, or cwd before filesystem writes occur.
- Isolated task runs produce at least one evidence ref, with `task-run://<runId>` always present.
- Task completion is not accepted solely from subagent prose; it is linked to runner evidence and later checked by the verification gates in [04 Verification Gates](./04-verification-gates.md).
